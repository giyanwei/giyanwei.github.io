HDFS

Hadoop Distributed File System (HDFs):分布式文件系统，用来存储海量数据。

单台机器性能纵向扩展的问题，是有硬件瓶颈的，包括成本也会指数型增长。所以需要分布式系统。

分布式处理数据的第一步就是将庞大的数据进行切块，分为多个block。Hadoop中这一步由HDFS来完成。默认数据块大小是128MB。分出的block再进行备份，总共会有三份备份，本机一份。

HDFS架构

分为namenode和datanode。

namenode工作职责：

  1. 管理各个datanode

  2. 管理文件信息，文件名、文件多大、文件被切块、存贮位置信息，即管理元数据信息。

  3. 基于RPC心跳机制控制集群中各个节点（datanode）的状态。

  4. namenode存在单点故障问题，可以启用一个备用namenode用来保证元数据信息安全。

  5. datanode挂掉后，数据丢失，因此需要控制datanode 的备份，默认3份，本机一份。

HDFS特性

HDFS优点

  1. 支持文件存储，TB、PB级别的数据，如每日的用户行为log

  2. 高容错性

  3. 横向扩展的花费呈线性

  4. 快速响应硬件故障

HDFS缺点

  1. 不适合类似数据库一样的低延迟访问数据，响应时间毫秒级别场景

  2. 不适合大量存储小文件

  3. 不适合多次修改数据，写入文件

Mapper进程

分布式处理数据的第一步就是将庞大的数据进行切块，分为多个block。MapReduce中，mapper只负责分发数据不负责具体的计算。Mapper进程会把被分到的block中的数据分发给所需的reduce。在hadoop中，有多少数据block，就会起多少mapper进程。可以一台机器上起多个进程。

MapReduce具体过程

Spliting阶段

InputSplit: 将文件进行分片，Map的输入为分片，每有一个输入分片会起一个mapper进程来处理。

默认情况下，输入片(InputSplit)的大小与数据块(Block)的大小是相同的，以HDFS的一个块的大小(默认128M)为一个分片，块的大小是可以设置。

大量小文件的性能问题

如果数据块(Block)的大小是默认值128MB，输入文件有两个，一个是32MB，一个是 172MB。那么小的文件是一个输入片，大文件会分为两个数据块，那么是两个输入片。一共产生三个输入片。每一个输入片由 一个Mapper进程处理。这里的三个输入片，会有三个Mapper进程处理。
这个例子展现了hadoop处理大量小文件的时候的性能问题：

  • hadoop处理大量小文件会产生过多的map任务。
  • 如果小文件过多，一个文件占用一个block， namenode存储这些元数据，这个会占用namenode的资源。
  • 访问大量小文件速度远远小于访问几个大文件。HDFS最初是为流式访问大文件开发的，如果访问大量小文件，需要不断的从一个datanode跳到另一个datanode，严重影响性能。

最后得出结论：Hadoop处理大量小文件速度远远小于处理同等大小的大文件的速度。

键值对

把文件的每一行文本内容解析成键值对。key是每一行的起始位置(单位是字节),value是本行的文本内容。<k, v> = <偏移量，一行文本>

Mapper阶段

对上个阶段解析出来的每一个键值对，调用一次map方法。如果有1000个键值对，就会调用1000次map方法。每一次调用map方法会输出零个或者多个键值对。

Shuffle阶段

shuffle过程包括在map方法输出（partition->sort>spill->merge）到reduce输入(merge->sort)

1. 暂存环形缓冲区

Mapper输出的键值对会暂时的放在一个环形的内存缓存区中（默认为100M, 有io.sort.mb属性控制)

2. Partition & Sort

在每个Mapper进程内部，将单个Mapper内所输出的一个个键值对按照键来分区。分区的方法可以通过自定义实现Partitioner接口类来指定。单个Mapper内的分区完成后，会对分区内部的数据进行一轮排序。（此时默认在内存内操作，数据量尚未大到溢出到磁盘）

3. Spill to Disk (Sort & Combiner & Merge)

  • 溢出到磁盘机制（溢写）。Spill可以认为是一个包括Sort和Combiner（Combiner是可选的，用户如果定义就有）的过程。
  • 缓存区快要溢时（默认为缓冲区大小的80%， 由io.sort.spill.percent属性控制）， 会在本地系统中创建一个溢出文件，将缓冲区中的数据写到这个文件中。
    • 可能会有许多的的溢出文件，这是需要将这些文件合并， 合并的过程中，会不断地进行排序和combiner操作， 目的有两个：
      • 尽量减少每次写入磁盘的数据量
      • 尽量减少下一复制阶段网络传输的数据量。最后合并成了一个已分区且已排序的文件。为了减少网络传输的数据量，这里可以将数据压缩，只要将mapred.compress.map.out设置为true就可以了

4. Copy (to reducer)

Reducer任务会主动从Mapper任务复制其输出的键值对，并且每个map传来的数据都是有序的。如果reduce端接受的数据量相当小，则直接存储在内存中；如果数据量超过了该缓冲区大小的一定比例，则对数据合并后溢写到磁盘中。
